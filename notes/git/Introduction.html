什麼是Git？

Git是一種分散式的版本控制系統，用於追蹤和管理軟體開發項目的原始碼，它允許多人協同工作，輕鬆地在不同的版本之間進行切換、合併和回溯。

版本控制

版本控制其實就是去紀錄一個檔案所修改的部分，以後想要去追蹤每個版本修改了什麼，或是當新版本有問題時，都可以快速的調閱紀錄以及回到某一個版本。

系統的種類

中央式系統

以一台伺服器為中心，所有的更新與操作都需要透過這個伺服器，也就是這個伺服器擁有絕對的控制權以及決策權，萬一這台伺服器故障了，或是沒有網路連線的環境，版本控制功能就沒辦法使用。

分散式系統

每台電腦都可以是一個節點，而每個節點都具有一定的控制權和決策權，可以獨立地進行操作和決策，即使伺服器故障或是沒有網路的環境下，還是可以用自己的電腦進行版本控制，之後等伺服器恢復正常運作或是在有網路的環境後再進行同步就好。

Git與GitHub的關係

Git和GitHub是兩個相關但不同的概念，開發者可以使用Git在本地進行版本控制，然後將其推送到GitHub上的遠端儲存庫，從而實現代碼的共享、協同工作和社區參與。

  • Git：一套免費、開源分散式的版本控制系統，用於管理和追蹤代碼的變更。
  • GitHub：一個基於Git的原始碼代管服務平台，提供了一個中央化的雲端儲存庫和協作工具。

Git的工作流程

  • 檔案（File）：檔案就是在本地電腦上創建或修改的專案文件，這些檔案可以是程式碼、設定文件、圖片…等。
  • 索引（Index）：索引也稱為暫存區（Staging Area），它是一個中介區域，用於存儲準備提交的更改，當修改或創建檔案後，需要將這些更改添加（add）到索引中，這樣Git才能夠追蹤這些更改。
  • 本地數據庫（Local Repository）：本地數據庫是存儲項目的歷史記錄和版本的地方，它可以創建新的也可以複製（clone）既有的遠端數據庫，並且會存放在專案的資料夾中，本地數據庫包含所有提交（commit）的歷史記錄、分支（branch）…等資訊，並且可以在本地數據庫中進行分支操作、合併（merge）…等操作。
  • 遠端數據庫（Remote Repository）：遠端數據庫是位於遠端伺服器上的Git數據庫，它可以是在GitHub、GitLab或其他Git托管服務器上建立的數據庫，遠端數據庫用於多個開發人員之間的協作和代碼共享，可以將本地數據庫的檔案推送（push）到遠端數據庫，也可以從遠端數據庫拉取（pull）最新版本的專案到電腦中。

什麼是分支？

分支（Branch）是指在數據庫中獨立開發和管理代碼的獨立指向，讓開發人員在同一個數據庫中同時進行不同的工作，並且可以將這些工作的進度和更改獨立保存。

使用分支有助於有效管理代碼庫，避免不同功能、修復或實驗性的更改相互衝突，開發人員可以根據需要創建和切換分支，使工作更有組織性和可追蹤性，分支也提供了一種實驗和測試新功能的方式，而不會影響主分支的穩定性。

    分支的主要特點包括：
  • 獨立性（Independence）：每個分支都是獨立的，擁有自己的提交歷史、文件狀態和更改。
  • 合併（Merge）：分支可以合併到其他分支，將不同的工作結果合併到一個統一的版本中。
  • 創建和切換（Create and Switch）：開發人員可以創建新的分支，並在不同的分支之間切換。
  • 同步（Synchronize）：分支可以用於隔離開發和測試，並在完成後將其與主分支同步。
  • 版本控制（Version Control）：分支提供了在不同的開發階段和功能之間進行版本控制的能力。
  • 合作（Collaboration）：多個開發人員可以在不同的分支上獨立工作，然後將它們的更改合併到共享的分支中。

分支合併

在Git中，要合併分支有兩種方法，分別是Merge與Rebase，它們有不同的工作流程和影響，但是都可以達到合併分支的效果。

Merge
    特色：
  • 創建一個新的合併提交，將兩個或多個分支的更改合併到一個共同的祖先上。
  • 保留原始分支的提交歷史，合併提交記錄了哪些分支進行了合併。
  • 不改變原始分支的提交歷史，保留分支獨立性。
    使用場景：
  • 在公共儲存庫或共享分支上進行合作開發。
  • 需要保留分支歷史，以便追蹤和查看每個分支的修改。

Rebase
    特色：
  • 將一個分支的提交應用到另一個分支上，重新應用提交以修改提交歷史。
  • 創建新的提交，形成線性提交歷史，看起來像一個連續的提交序列。
  • 使得合併後的分支更加整潔，減少合併提交的數量。
    使用場景：
  • 在自己的私有分支上進行個人開發，整理提交歷史以保持清晰和一致。
  • 將特性分支合併到主分支上，以獲得線性提交歷史和更好的可讀性。

合併衝突

合併衝突（Merge Conflict）是在合併過程中發生的一種情況，當兩個分支的同一個檔案的同一個位置有不同的修改時，Git無法自動決定應該保留哪個版本的更改，因此產生衝突，當Git檢測到合併衝突時會在衝突檔案中標記衝突的位置。

    解決合併衝突的步驟：
  1. 檢查衝突：在執行合併操作後，Git 會標記衝突的檔案，你可以使用git status指令來檢查哪些檔案出現了衝突。
  2. 解決衝突：仔細檢查衝突的內容，根據專案需求選擇保留哪個版本的更改或進行修改，在衝突檔案中手動編輯，將兩個版本的內容合併成一個正確的結果。
  3. 刪除衝突標記：在衝突解決完畢後，刪除衝突標記和不需要的內容，僅保留所需的正確內容。
  4. 儲存檔案：保存衝突解決後的檔案。
  5. 標記衝突已解決：使用git add命令將已解決的衝突標記為已解決，表示這些部分已經確定合併結果。
  6. 完成合併提交：使用git commit命令提交合併結果，Git會自動創建一個合併提交，其中包含解決衝突的更改。
